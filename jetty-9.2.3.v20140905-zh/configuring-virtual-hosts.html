一个虚拟主机是一个可选择的名称,在 DNS 中注册,用于一个 IP 地址,这样多个域名将被解析到一个共享的服务器实例的同一个 IP. 如果要服务的内容的别名名称是 不同的(If the content to be served to the aliases names is different), 那么每个被部署的 context 需要配置一个虚拟主机来表明哪个 context 将作出响应.

虚拟主机被设置到一个 context 通过调用 setVirtualHosts 或 addVirtualHost 方法来完成.

  • 在 webapps 目录中(见 jetty 分发包中的 test.xml 示例 )使用一个 context XML 文件.

  • 使用 WEB-INF/jetty-web.xml 文件(废弃).

  • 使用一个 binding 创建一个 自定义的 deployer 为同一个 webapps 目录下的所有的 contexts 配置虚拟主机.

  • 直接在 嵌入式 使用上调用 API.

虚拟主机名称:

Jetty 支持以下风格的虚拟主机名称:

www.hostname.com

一个完全限定主机名称.列出所有的变体很重要,因为一个站点既可以从 www.hostname.com 接收传输也可以只是从 hostname.com 接收传输.

*.hostname.com

一个通配符限定主机,将只匹配任意名称中的一个级别. *.foo.com 将匹配 www.foo.com 和 m.foo.com, 但是 www.other.foo.com 不行.

10.0.0.2

一个 IP 地址可以被给定作为一个虚拟主机名称表明一个 context 应该处理在那台服务器端口上接收到的请求,但是没有一个主机名称被指定(HTTP/0.9 或 HTTP/1.0)

@ConnectorName

一个 connector 名称,不是一个严格的虚拟主机,但是只会匹配一个通过 Connector.setName(String) 设置有一个匹配名称的 connectors 上接收到的请求,

www.√integral.com

非 ascii 和 IDN 域名可以被使用 Puny Code 被设置为虚拟主机,等同于可以从一个 Punycode/IDN 转换器 获得. 比如如果非 ascii 域名 www.√integral.com 给定到一个浏览器中,然后它将使用域名 www.xn--integral-7g7d.com 作出一个请求,名称应该被加上作为虚拟主机名称.

使用不同的虚拟主机来选择不同的 Contexts

你可以通过提供一个特定的虚拟主机列表来配置不同的 contexts 在不同的虚拟机上作出响应.

比如,假设你的虚拟机上有这些 DNS 名称:

  • www.blah.com

  • www.blah.net

  • www.blah.org

  • www.other.com

  • www.other.net

  • www.other.org

假设你有2个 webapps,一个为 blah.war,你想从*.blah.* 名称中得到服务, 一个为 other.war,你只想从 *.other.* 名称中得到服务.

使用不同的虚拟主机在相同的 context path 上选择不同的 Contexts.

在前一个小节,我们设置了2个不同的 contexts 在 不同的 context paths 上从不同的虚拟主机获取服务. 然而,没有要求 context paths 必须是不同的:你可以对多个 contexts 使用相同的 context path,并使用虚拟主机来决定哪个服务于一个给定的请求.

考虑一个示例,我们有和之前相同的一组 DNS 名称,和相同的 webapps blah.war 及 other.war. 我们想让 blah.war 在 *.blah.* 的主机名称上作出响应, 并且 other.war 在 *.other.* 名称上作出响应. 然而,我们想让我们 所有的 客户端使用 "/" context path, 无论指向的 context 是哪一个.

换句话说,我们想让下面所有的 urls 映射到 blah.war:

  • http://www.blah.com/

  • http://www.blah.net/

  • http://www.blah.org/

类似地,我们想让下面所有的 urls 映射到 other.war

  • http://www.other.com/

  • http://www.other.net/

  • http://www.other.org/
